## 1. HTML（可背版）

### 30秒

HTML 核心是“语义化 + 可访问 + SEO 友好”。语义化标签如 `header`、`main`、`section` 能提升结构清晰度；`label` 绑定输入框、图片加 `alt` 能提升无障碍；`title`、`description`、标题层级能帮助 SEO。

### 90秒

我写 HTML 会先搭语义骨架：`header/nav/main/footer`，再补可访问属性，比如表单 `label for`、按钮语义、导航 `aria-label`。SEO 方面优先补齐 `title`、`description`、`h1-h3` 层级和图片 `alt`。这样页面后续接 CSS 和 JS 时可维护性更高，搜索收录和读屏体验也更稳定。

### 高频追问一句话

- 为什么语义化重要：结构清晰、无障碍、SEO 三赢。
- 为什么少用 `div`：语义信息丢失，维护和可访问性都变差。

---

## 2. CSS（可背版）

### 30秒

CSS 我按“布局优先、响应式、性能”来写：布局优先 `flex/grid`，统一 `box-sizing: border-box`，移动端用媒体查询；动画优先 `transform/opacity`，尽量减少重排重绘。

### 90秒

我一般先定设计变量，再做布局：外层 `grid` 控区块，内部 `flex` 控对齐。响应式用断点把双列降成单列。性能上避免频繁改几何属性，动画优先 `transform`。如果页面卡顿，会先查强制同步布局和大面积重绘，再做样式收敛。

### 高频追问一句话

- BFC 作用：防止 margin 折叠、清除浮动、做自适应布局。
- `position: absolute` 参考谁：最近的非 `static` 祖先。

---

## 3. JavaScript（可背版）

### 30秒

JS 我重点抓“数据流 + 事件流 + 渲染流”：状态集中管理，事件委托减少监听器，更新后统一渲染，避免散落式 DOM 操作。

### 90秒

做交互时我会先定义状态源，比如 `tasks` 数组；然后处理输入事件，更新状态后只走一个 `render()`。这样好处是逻辑可追踪、可测试。性能层面会用事件委托、防抖节流；稳定性层面会处理空值、异常和边界条件。

### 高频追问一句话

- 事件循环：同步代码 -> 微任务 -> 渲染 -> 下一个宏任务。
- `var` 循环 + `setTimeout` 问题：同一个函数作用域，最终读到同一个值。

---

## 4. React `useState`（可背版）

### 30秒

`useState` 用于组件局部状态。依赖旧值更新时必须用函数式写法 `setX(prev => ...)`，避免并发更新下读到旧值。

### 90秒

我用 `useState` 管输入框、开关、计数器这类局部状态。更新对象或数组时一定返回新引用，不能原地修改。若状态很多，我会按“变化频率和职责”拆分，减少不必要重渲染。

### 高频追问一句话

- 为什么不用直接赋值：React 需要新引用触发更新。
- 什么时候用 `useReducer`：状态复杂、动作多、迁移成本高时。

---

## 5. React `useEffect`（可背版）

### 30秒

`useEffect` 专门处理副作用：请求、订阅、定时器、DOM 操作。核心是依赖数组和清理函数，防止重复执行和内存泄漏。

### 90秒

我会把副作用放在 `useEffect`：初始化请求用 `[]`，依赖变化用 `[dep]`。监听器和定时器必须在返回函数里清理。请求场景要处理竞态，比如取消标记或中断信号，避免组件卸载后还写状态。

### 高频追问一句话

- 依赖数组不写：每次渲染后都执行。
- 为什么会死循环：`effect` 里更新了它自己依赖的值。

---

## 6. React `useContext`（可背版）

### 30秒

`useContext` 适合低频全局共享状态，如主题、语言、登录信息，解决多层传参问题。

### 90秒

我会先 `createContext`，再在顶层 `Provider` 注入 `value`，子组件通过 `useContext` 读取。注意 `Provider` 的 `value` 变化会让消费者重渲染，所以高频复杂状态不建议只靠 Context，通常配合 Zustand/Redux。

### 高频追问一句话

- Context 能替代状态库吗：小项目可以，大型复杂状态不建议。
- 怎么优化 Context 重渲染：拆分 Context、稳定 `value` 引用。

---

## 7. Zustand（可背版）

### 30秒

Zustand 是轻量全局状态库，不需要 `Provider`。核心是 `create` 定义 `state + action`，组件用选择器订阅，减少无关重渲染。

### 90秒

我会把业务共享状态放进 Zustand，例如列表、筛选、用户会话。Store 中集中写 action，组件只负责展示和触发。订阅时写 `useStore(s => s.xxx)`，避免整仓订阅。持久化场景用 `persist` 中间件，把主题、用户偏好存到本地。

### 高频追问一句话

- Zustand 和 Redux 区别：样板代码更少，上手更快。
- 为什么推荐选择器：精确订阅，性能更稳。

---

## 8. 四者分工（面试高频）

### 30秒

分工是：局部状态用 `useState`，副作用用 `useEffect`，低频跨层共享用 `useContext`，复杂全局共享用 Zustand。

### 90秒

我会按状态范围和更新频率做分层：组件内临时态放 `useState`；请求、监听、计时器放 `useEffect`；主题语言这类低频全局放 `useContext`；跨页面且更新复杂的业务状态放 Zustand。这样可维护性和性能都更平衡。

---

## 9. 高频追问速答（可直接背）

1. `useEffect` 和 `useLayoutEffect` 区别？

   - 前者在渲染后异步执行，后者在绘制前同步执行。
2. Zustand 为什么快？

   - 选择器精确订阅 + 状态更新链路短。
3. 什么时候别用 Context？

   - 高频更新、数据规模大、跨模块复杂联动时。
4. 如何避免 React 重渲染过多？

   - 状态下沉/拆分、`memo`、选择器订阅、稳定引用。
5. 页面卡顿先查什么？

   - 长任务、频繁重排重绘、无节制事件回调。

---

## 10. 1分钟总复述（终极背诵）

前端实现我通常按四层做：HTML 先语义化和可访问，CSS 先布局再响应式并注意渲染性能，JS 用状态驱动渲染和事件委托。到 React 里，局部状态用 `useState`，副作用用 `useEffect` 并做好清理，跨层低频共享用 `useContext`，复杂全局业务状态用 Zustand。这样代码结构清晰、性能可控、面试也好解释。
